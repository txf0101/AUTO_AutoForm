# 01 项目全景与目录

## 项目一句话

本项目把本机 AutoForm Forming 的常见工作流整理成可测试的 Python 能力、可选 MCP 工具、浏览器工作台和多 Agent 调度骨架。维护者要关心三件事：事实来源是否可靠，危险动作是否被批准，结果是否能复核。

## 顶层目录

| 路径 | 给维护者的解释 |
| --- | --- |
| `autoform_agent/` | Python 主包。项目的大部分业务逻辑、CLI、HTTP bridge、Agent runtime、MCP wrapper 和 AutoForm 工具能力都在这里。 |
| `autoform_agent/flex_scripts/` | 柔性脚本层。包含 Script Agent、Script Executor、registry、sandbox、验证器和 CAD 实测能力。 |
| `frontend/` | 本地浏览器工作台。用户在这里输入 prompt、配置 provider、观察 Agent 图谱、终端输出、token 用量和 API 状态。 |
| `tests/` | Python 测试。每个业务模块都应该有对应测试，危险动作默认不执行也要有测试。 |
| `frontend/tests/` | 前端烟雾测试。用于确认页面骨架、关键 data 属性、九个 Agent 图谱和脚本入口没有被破坏。 |
| `docs/` | 项目常规开发文档。包含新手上手、调用链、架构、RAG、执行器、维护检查清单等。 |
| `VC开发文档/Auto_Autoform思路整理/` | 项目开发思路来源目录。后续改需求、架构、代码、测试、README、入门文档或复盘前，先读这里的相关 DOCX。 |
| `schemas/` | P0 事件、任务、补丁、证据、token 等结构契约。前后端字段变更要优先核对这里。 |
| `flex_script_library/` | 稳定脚本库。每个 SkillCard 的版本、入口脚本、样例和后续入库记录都应在这里沉淀。 |
| `scripts/flex/` | 柔性脚本共享辅助目录。当前只放共享说明和后续脚本侧工具。 |
| `fixtures/` | 固定回放数据。前端和后端可以在无真实求解器时复现一条低风险流程。 |
| `policy/` | 权限和风险边界。高风险动作、字段写入、明文 key 等规则应在这里有依据。 |
| `evals/` | 评测用例。用于描述端到端场景或后续评估问题。 |
| `handoff/` | 每轮开发复盘和交接。这里记录为什么改、怎么验证、还有哪些风险。 |
| `output/` | 本地运行输出。包括工程运行副本、日志、截图、报告包和启动器记录。 |
| `output/script_runs/` | ScriptRunRecord、stdout/stderr、验证报告和脚本证据目录。 |
| `output/cad_measurements/` | CAD 实测结果与解析器探测证据。STEP/IGES completed、blocked 或 failed 结果都要落盘。 |
| `tmp/flex_script_sandbox/` | fork、新建、patch 和 validate 的临时脚本空间。稳定库不能在这里被原地覆盖。 |
| `维护者入门阅读文档/` | 本文档夹，面向维护者和开发者的项目阅读入口。 |

## Python 主包分区

| 路径 | 职责 | 维护提醒 |
| --- | --- | --- |
| `autoform_agent/agent_runtime.py` | 页面和 CLI 的后端 Agent runtime。它合并配置、构造中心 Agent 计划、执行白名单工具、调用 provider 并返回 JSON。 | 不要在这里泄露明文 key。新增本地能力时，先确认是否应进入 `_runtime_tool_registry()` 或走 `AgentToolGateway`；审批阻断动作要写入 `executionContext.pending_approval` 和 `resumable_action`。 |
| `autoform_agent/http_bridge.py` | 本地 HTTP 适配器，接收 `/api/agent` 请求。 | 这里应保持薄封装，不承载复杂业务判断。 |
| `autoform_agent/agent_system/` | 多 Agent 契约、角色注册、中心 Agent 内核、路由预览、实时执行器和工具网关。 | 改角色、路由、事件或网关权限时，测试和前端说明也要同步。 |
| `autoform_agent/mcp_server.py` | 可选 MCP stdio 入口。 | MCP host 用它直接调用工具；网页主链路不依赖外部 MCP host。 |
| `autoform_agent/mcp_tools/` | MCP wrapper。每个模块把 MCP 参数转换为业务函数输入。 | wrapper 应薄，业务规则放到主业务模块。 |
| `autoform_agent/project_workflow.py` | 工程解析、复制、打开 GUI、求解计划、结果清点和报告包组织。 | 复制、打开窗口、求解要分清；用户只要求打开窗口时不要顺手提交求解。 |
| `autoform_agent/paths.py` | AutoForm 安装发现、ProgramData 和示例目录推导。 | 跨机器适配先改这里，并补安装发现测试。 |
| `autoform_agent/inventory.py` | 示例工程和 AFD 项目摘要。 | 只读事实优先放这里，便于 runtime、CLI 和 MCP 共用。 |
| `autoform_agent/solver.py` | 求解器命令计划、探测和日志摘要。 | 默认保持计划或 dry run；真实求解要有明确参数和测试边界。 |
| `autoform_agent/results.py` | 结果证据清点、轻量报告包和结果目录扫描。 | 结果判断要可追溯到文件、日志或清单。 |
| `autoform_agent/result_viewer.py` | V1.1 GUI 后处理语义、视角映射、证据计划和阻塞项。 | 自动窗口操作必须尊重控件证据和执行批准。 |
| `autoform_agent/gui_automation.py` | Win32 窗口快照、截图、焦点和低层交互辅助。 | 只在用户明确要求真实 GUI 操作时使用。 |
| `autoform_agent/preparation_agents.py` | 低风险准备链路，生成需求、几何、材料、工艺、脚本和阶段摘要。 | 输出保持候选状态，不直接写正式工程结论。 |
| `autoform_agent/flex_scripts/` | 受控脚本生产和执行层。稳定脚本来自 `flex_script_library`，临时脚本只能写入 `tmp/flex_script_sandbox`，运行记录写入 `output/script_runs`。 | 新脚本必须有 SkillCard、验证报告和 ScriptRunRecord；L2 入库要绑定静态审计、依赖探测、审批记录和验证报告 hash；STEP/IGES 只有 parser 完成时才写实测尺寸。 |
| `autoform_agent/report.py` | 报告、模板和交付计划相关能力。 | 报告内容必须绑定来源和结果证据。 |
| `autoform_agent/release.py` | 发布就绪检查。 | 对外发布前用它核对 README、许可、环境、MCP 配置和测试入口。 |
| `autoform_agent/safety.py` | 公开发布扫描和敏感信息检查。 | 增加新文件类型时，同步扩展扫描范围。 |

## 前端分区

| 文件 | 职责 | 维护提醒 |
| --- | --- | --- |
| `frontend/index.html` | 页面骨架。所有可被脚本绑定的区域都有 `data-*` 属性。 | 不要随意删 `data-*`，测试依赖这些稳定钩子。 |
| `frontend/styles.css` | 视觉样式。九个 Agent 使用稳定三列布局。 | 绿色只表示运行中，完成后应回灰白待命。 |
| `frontend/app.js` | 页面状态、fixture 回放、HTTP 请求、Agent 图谱、终端输出和 API 面板。 | 页面不直接决定 AutoForm 工具，只把 prompt、批准状态和压缩后的 `execution_context` 交给后端。 |
| `frontend/README.md` | 前端入口和维护说明。 | 改 bridge、端口、图谱、API 面板或本机执行选项时同步更新。 |

## 多 Agent 的当前业务视图

前端只展示九个业务 Agent：

1. 中心Agent
2. 需求与工艺规划Agent
3. 几何与数据Agent
4. 材料Agent
5. 工艺设置Agent
6. 求解执行Agent
7. 后处理Agent
8. 诊断与优化Agent
9. 报告整理Agent

内部历史角色仍然存在，例如 `manager`、`project_workflow`、`solver`、`result_review`、`reporting`。这些角色通过映射进入九个业务节点。这样做的目的很实际：旧测试和工具链继续可用，用户看到的图谱保持稳定。

## 维护时最容易误解的地方

- MCP 有两种含义：外部 MCP host 可以启动 `mcp_server.py`；内部 Agent runtime 也能复用 MCP wrapper 中的同源工具函数。网页主链路走后端 runtime，不需要外部 MCP host 才能工作。
- `execute=False` 不等于什么都不做。若 `open_gui=True` 且网关已经批准，项目可以复制并打开窗口，但求解器仍保持不执行。
- `done` 和 `completed` 在前端图谱里会回到待命灰白态。绿色只代表当前正在工作。
- 专业 Agent 输出多为候选建议。正式写入或真实执行要经过中心 Agent、网关、审查和批准。
- 材料 Agent 的本机材料库检索通过 `skill_material_database_query` 形成 `ScriptRunRecord`；材料来源和弹性常数分别通过 `skill_material_source_candidate_set`、`skill_material_elastic_constants_candidate_set` 形成候选记录。前端跨轮只回传压缩 `conversationContext`，不要把完整命令输出塞回模型上下文。
- 审批阻断不是错误摘要。后端应保留 `pendingApproval`、`resumableAction`、原始工具参数和证据目录；用户批准后按同一 `task_id` 继续，不重新解析上一轮中文摘要。
- CAD 文件名中的 `30-40-3` 只能作为候选尺寸。CadQuery/OCP、FreeCADCmd 或内置 STL parser 产出的 bbox 才能进入 `cad_measurement_result.length/width/thickness`。
